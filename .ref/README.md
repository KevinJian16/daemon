# .ref 文档总则

## 0. 这份 README 的作用

这是 `.ref/` 的入口文档。

它只负责四件事：

1. 说明当前**五份权威文档**各自做什么
2. 规定文档之间的权威顺序
3. 记录当前重构工作的推进顺序
4. 说明审计文档和历史归档文档的角色

无论是实现者还是审计者，都应先读本文件。

---

## 1. 当前五份权威文档

当前正式权威文档只有五份：

1. `TERMINOLOGY.md`
2. `INTERACTION_DESIGN.md`
3. `DESIGN_QA.md`
4. `daemon_实施方案.md`
5. `README.md`

`config/lexicon.json` 是术语词典的**机器可读数据源**，但它不是第六份权威说明文档；它受 `TERMINOLOGY.md` 约束，并服务于 Console 词典和术语显示。

### 1.1 各文档职责

| 文档 | 职责 | 主要使用者 |
|---|---|---|
| `README.md` | 文档治理入口。说明权威关系、推进顺序、审计定位、归档规则 | 实现者 / 审计者 |
| `TERMINOLOGY.md` | 术语权威。规定正式对象、中文显示名、英文 canonical name、旧新机制映射 | 实现者 / 审计者 |
| `INTERACTION_DESIGN.md` | 交互权威。规定 Portal / Console / Telegram 的对象语义、行为语法、风格与边界 | 实现者 / 审计者 |
| `DESIGN_QA.md` | 细节确认文档。记录已确认的机制细节、边界和例外；与实施方案冲突时以 QA 为准 | 实现者 / 审计者 |
| `daemon_实施方案.md` | 全机制说明书与实施规范。给出对象模型、事件流、存储、执行、学习与界面映射的正式落地方案 | 实现者 / 审计者 |

## 2. 当前机制的正式对象

当前正式事项机制已经切换为：

- `Draft`
- `Slip`
- `Folio`
- `Writ`
- `Deed`

这意味着：

1. `Slip` 是任务载体，`Deed` 是执行实例
2. `Folio` 是正式主题容器
3. 所有事项先成一张 `Slip`，超限再开 `Folio`
4. Portal 与 Console 都要服从同一套对象机制与同一套交互语法

在这一轮重构里，任何过时残余都应被消除，不与当前机制并存。

---

## 3. 权威顺序

五份权威文档的职责不同，但它们共同构成当前的正式依据。

冲突处理顺序如下：

1. 术语冲突：以 `TERMINOLOGY.md` 为准
2. 交互冲突：以 `INTERACTION_DESIGN.md` 为准
3. 机制细节冲突：以 `DESIGN_QA.md` 为准
4. 实施细节冲突：以 `daemon_实施方案.md` 为准
5. `README.md` 只负责解释这种权威关系，不单独定义机制细节

`config/lexicon.json` 只承担术语数据源角色，不单独推翻或新增术语定义。

## 4. 当前工作顺序

当前重构顺序已经确定，不再按旧 Task 2 / 3 / 4 的边界推进。

### 第一阶段：先重写权威文档

先把新机制完整写进五份权威文档，而不是停留在讨论稿或额外草案中。

### 第二阶段：再重构后台对象模型

把系统现实改成：

- `Draft`
- `Slip`
- `Folio`
- `Writ`
- `Deed`

并清除过时对象分层、过时命名和过时任务分类的主机制地位。

### 第三阶段：先做 Portal，再做 Console

Portal 先落：

- `草稿 / 签札 / 卷` 的前台对象
- 统一的对象机制
- 统一的交互语法
- 新的前台风格与空间组织

Console 后落：

- 与新对象机制对齐的治理面
- 全中文正式术语显示
- 词典、状态、运维和结构视图

### 第四阶段：补学习与自动化闭环

等对象模型和界面都稳定后，再补：

- `Draft / Slip / Writ` 的规划学习
- standing `Slip` + repeated `Deed`
- `Writ` 的学习与候选修订

### 第五阶段：删旧、回归、接受审计

最后统一做：

- 删除过时残留
- 删除已失效的旧接口和旧文案
- 回归验证
- 交给审计方根据新的五份文档重写并维护 `MECHANISM_AUDIT.md`

## 5. 审计文档的角色

`MECHANISM_AUDIT.md` 不是当前五份权威文档之一。

它的角色是：

- 根据五份权威文档重写 gap 检查框架
- 在实现推进后检查实现与文档是否一致
- 维护跨层机制审查项

它不是：

- 机制说明书
- 交互设计文档
- 术语权威
- 实施规范

审计者应先读本 README，再读五份权威文档，然后据此维护 audit。

## 7. 当前提醒

当前最容易出错的地方有四个：

1. 把 `Deed` 重新混回“任务本体 + 执行实例”的双重角色
2. 把 `Folio` 做成普通列表容器，而不是正式主题对象
3. 重新引入多套并列任务类型作为主机制
4. 把 Portal / Console 各自做成一套不同的对象世界

另一个高风险误区是把前端风格理解成“可选偏好”。

当前正式约束是：

- Portal = Claude 静态 + Apple app 层级动态 + 文件系统式对象组织
- Console = Claude 静态 + Apple app 层级动态

这条约束属于正式交互机制的一部分，不是视觉润色建议。

如果发现机制定义冲突，优先修正文档权威的一致性，再修代码。
