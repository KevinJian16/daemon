# Daemon 交互设计

> 日期：2026-03-10  
> 本文档定义 Portal、Console、Telegram 三个表面的交互哲学、对象语义、层级结构与行为规则。  
> 机制对象以 `TERMINOLOGY.md` 与 `daemon_实施方案.md` 为准；本文只规定这些对象如何被看见、被操作、被组织。

---

## 0. 总原则

### 0.1 静态学 Claude

静态设计目标不是“参考 Claude”，而是**尽量直接继承 Claude 网页端那套内容产品静态语言**。

允许适配 Daemon 自身对象，但不允许退化成普通 web app 的后台布局。

Portal 与 Console 在静态层面共同遵循：

- 内容先于控件
- 排版即层级
- 留白克制
- 色彩只在有语义时出现
- 页面像成熟内容产品，不像后台工具、工单系统或任务管理器
- 不堆按钮、不堆 badge、不堆面板、不靠卡片阵列建立主结构
- 主层级靠字重、段落、间距、密度、边界和滚动节奏建立
- 主视图先像“可阅读对象”，再像“可操作系统”

### 0.2 动态学 Apple

动态设计目标不是“有一点 Apple 风味”，而是**所有会动的东西都尽量使用 Apple 的动态语言**。

它的核心不是动画装饰，而是空间与物感：

- 层级切换有明确方向
- 对象进入、展开、收束有连续感
- 平时极简，操作时才显露结构
- 所有转场都帮助用户理解“我从哪里来、现在在哪里、接下来会到哪里”
- 动效节奏要克制、精确、有弹性，不做普通 web 动画味
- 不允许硬切、瞬切、突兀弹出、无来处的状态闪现

### 0.3 Portal 与 Console 的动态分工

这条必须明确区分：

- **Portal**
  - 页面层级动态：采用 Apple 原生 app 式层级转场
  - 对象组织动态：采用文件系统式对象操作感
- **Console**
  - 页面层级动态：采用 Apple 原生 app 式层级转场
  - 不采用文件管理器味的对象组织语言

更准确地说：

- Portal = Claude 静态 + Apple app 层级动态 + 文件系统式对象组织
- Console = Claude 静态 + Apple app 层级动态

### 0.4 统一的机制

统一的机制不是统一皮肤，而是统一：

- 对象
- 状态
- 动作
- 手势语义
- 结果呈现

Portal 与 Console 不是两套不同现实，只是同一系统的两种观看方式。

两者都必须服从同一组正式对象：

- `Draft`
- `Slip`
- `Folio`
- `Writ`
- `Deed`

### 0.5 统一的交互语法

以下交互语义在 Portal 与 Console 里必须保持稳定：

- **点开某项**：进入该对象的下一层
- **下拉**：刷新当前视图对应的现实状态
- **返回**：退回上一层
- **长按 / 拖起**：进入对象操作态
- **拖放到容器或落点**：改变对象关系或对象姿态

同一种动作不能在不同页面表达不同类型的意图。

### 0.6 两类 Apple 动态语言

为了避免误读，这里明确区分两种都属于 Apple 的动态语言：

1. **原生 app 层级动态**
   - 用于列表 -> 详情 -> 子详情
   - 用于进入、返回、展开、收束、下拉刷新、sheet / side detail
   - Portal 和 Console 都必须使用

2. **文件系统式对象动态**
   - 用于拖拽、吸附、让位、成组、入卷、出卷、重排
   - 只在 Portal 中作为主要组织语言使用
   - Console 原则上不使用这种“桌面整理”式动效

## 1. 正式前台对象

### 1.1 `Draft`

`Draft` 是草稿态对象。

它表示：

- 一件事还在成札前的收敛阶段
- 计划仍在被修改
- 事件或规则生出的候选事项尚未正式成札

`Draft` 不是散乱的暂存聊天，而是真实对象。

### 1.2 `Slip`

`Slip` 是最小前台任务对象。

所有真正成立的“一件事”，都先以 `Slip` 的方式出现。

每张 `Slip` 都必须拥有：

- 标题
- 自己的对话页
- plan card
- 结果区
- 历史 `Deed`

### 1.3 `Folio`

`Folio` 是卷，是收纳并组织若干 `Slip` 的容器。

它不是大号 `Slip`，也不是大号列表页。

在前台，它是：

- 一种组织关系
- 一种主题对象
- 一种可打开、可收纳、可重排的卷面

### 1.4 `Writ` 与 `Deed` 的可见性

`Writ` 和 `Deed` 也是正式对象，但不应在前台抢占高频组织层。

- `Writ` 主要在 Console 中明确呈现
- `Deed` 主要在 `Slip` 页中作为这张签札的一次次行事记录被看见

## 2. Portal

### 2.1 Portal 的角色

Portal 是主人和 Daemon 共用的一张案桌。

它不是：

- 任务管理器
- 工单后台
- 流程编排器

它是：

- 草稿收敛的入口
- 签札的对话页
- 卷的组织面
- 结果与反馈回流的主界面

Portal 必须同时满足三件事：

1. 静下来时像 Claude 那样安静、克制、内容先行
2. 打开、返回、展开、收束时像 Apple 原生 app
3. 整理 `Slip / Folio` 时像文件系统那样操作对象

### 2.2 核心范式

Portal 的两个核心页面是：

- `Slip` 对话页
- `Folio` 卷页

`Slip` 是具体一件事的页面。  
`Folio` 是把若干相关 `Slip` 收为一卷后的父层页面。

### 2.3 `Slip` 页面

`Slip` 页面承载任务定义层。

内容：

- DAG 展示
- 统一对话流（承载 DAG 调整、执行调整、评价反馈等所有交互）
- 内嵌 Deed 执行块（可展开，显示 DAG 进度、Offering 版本和执行细节）
- 历次 Deed 列表（旧 Deed 逐渐淡去）
- 动作按钮区（按触发类型动态显示，见下文）

`Slip` 页面只有一条对话流，承载所有交互：调整 DAG、执行期间的调整和评价、收束后的反馈。对话统一在 Slip 层，Deed 没有独立对话。

**按触发类型动态显示按钮**（三种触发类型互斥，见 `DESIGN_QA.md` Q8.5）：

| 触发类型 | Slip 页面显示 |
|---------|-------------|
| 手动触发 | 「执行」按钮（创建 Deed + 立即运行，原子操作） |
| 定时触发 | 定时设置信息（下次触发时间、周期等） |
| 前序事件触发（Writ chain） | 显示等待的前序条件（如"等待 Slip A 收束"）；条件未满足时按钮禁用 |

`Deed` 也可由定时、Writ 联动等非用户事件产生。「执行」= 创建 Deed + 立即运行（原子操作，不可拆分）。

因此：

- `Slip` 是持续对象（任务定义）
- `Deed` 是这张 `Slip` 下的一次次具体执行（执行实例）
- 对话统一在 Slip 层，Deed 作为内嵌执行块存在于 Slip 对话流中

### 2.3.1 `Deed` 执行块

`Deed` 不是独立页面，而是内嵌在 Slip 对话流中的可展开执行块。执行后自动出现在对话流中。

执行块内容：

- 当前执行在 DAG 上的进度
- Offering 版本和执行细节
- 按钮：开始/停止（toggle）、收束

执行期间的调整和评价通过 Slip 统一对话流进行。running 期间对话不影响当前执行。

Deed 内多次开始/停止是同一个 Deed，不创建新 Deed。

### 2.3.2 `Deed` 收束后

Deed 收束后执行块冻结为只读：

- 产物标签
- 执行细节
- 无操作按钮

过去的 Deed 执行块保留但有保质期，过期从对话流中淡去。

### 2.4 `Folio = 卷页`

`Folio` 页回答四件事：

1. 这卷最近在推进什么
2. 里面有哪些 `Slip`
3. 哪些 `Slip` 正在活跃
4. 最近收束出了什么结果

它不应只是二次堆列表，而应有明确的卷面结构。

### 2.5 所有 `Slip` 都有 plan card

plan card 是统一原件，不因事情简单而消失。

规则：

1. 每张 `Slip` 都必须有 plan card
2. 简单事项的 card 更短、更紧
3. 复杂事项的 card 更长、更深
4. 不允许因为轻重不同而变成不同 UI 物种

### 2.6 前台对象只有 `Slip` 与 `Folio`

前台正式对象只有：

- 单张 `Slip`
- 一卷 `Folio`

单 `Slip` 之外的复杂事项，直接提升为 `Folio`。

### 2.7 Portal 的控制原则

Portal 不使用控制台式按钮阵列。

控制分成两类：

1. **语义控制**
   - 改目标
   - 改 plan
   - 追加要求

   这类通过对话表达。

2. **结构与姿态控制**
   - 入卷 / 出卷
   - 归并成卷
   - 重排
   - 收起 / 续办 / 暂放

   这类通过直接操作表达。

因此 Portal 的主要控制不应长成：

- 一排主按钮
- 工具栏式任务控制区
- 固定 slot 面板

而应长成：

- 对话语义
- 长按 / 拖起后的对象操作态
- 有空间感的落点、吸附、成组、入卷、出卷

### 2.7.1 按钮-对话等价原则

按钮和对话是同一控制通道的两种表达形式，系统对两者一视同仁：

- **按钮** = UI 便利，用于明确的状态边界动作（执行、停止、收束）。按钮点击在对话流中生成等价记录。
- **对话** = 自然语言交互，可触发状态变更（如 rework）。counsel 自主判断对话意图。

两者不是分离的控制通道，而是完全等价。按钮生成的记录和用户输入的消息在系统中被同等处理。

按钮的独特作用在于**回溯定义边界**：按下哪个按钮，回溯决定前面对话段的分类和去向。

### 2.7.2 操作-自然语言对应

所有用户所做的非对话框操作（按钮点击、拖拽、姿态变更等），都在对话流中生成对应的自然语言记录。对话流是完整的行为日志。

因此 **Slash command 已取消**。所有结构控制由按钮或直接操作承担，对话框只承载自然语言对话。

### 2.7.3 Slip 链 DAG 导航

Folio 内的 Slip 通过 Writ 连接形成 DAG。Slip 页面支持链内前后导航：

- **线性链**：单个"上一张"/"下一张"导航标签
- **分支点**（当前 Slip 有多个后继）：多个"下一张"标签，每个标注目标 Slip 名称
- **合并点**（当前 Slip 有多个前驱）：多个"上一张"标签，每个标注来源 Slip 名称

导航标签动态反映 Writ DAG 结构。这是 §2.6 Folio 内兄弟导航的扩展——兄弟导航处理线性排列，DAG 导航处理 Writ 依赖关系。

### 2.8 Portal 的空间语法

拖拽是 Portal 的正式语言之一。

拖拽承担两类含义：

1. **组织关系**
   - `Slip -> Folio`
   - `Slip -> Slip`，生成新 `Folio`
   - `Slip` 在 `Folio` 内重排

2. **对象姿态**
   - 当对象进入操作态时，界面可显露有限落点
   - 拖到这些落点，表示改变该对象当前姿态

当前文档只规定这是一套空间语法，不规定最终文案必须是哪几个按钮或词。

这里的动态语言要尽量贴近：

- iOS 主屏幕 app 的拖拽成组
- Files / Finder 的对象进入容器
- Apple 系统里对象被抓起、让位、吸附、归位的感觉

禁止把它做成：

- Kanban 看板拖拽味
- HTML5 demo 式拖放味
- 普通 web dashboard 的排序味

### 2.9 两张 `Slip` 合成一卷

当一张 `Slip` 被拖到另一张 `Slip` 上时：

1. 系统自动创建新的 `Folio`
2. 系统自动给出默认标题
3. 标题允许轻量改名

行为应类比 iOS 把一个 app 拖到另一个 app 上形成文件夹。

### 2.10 Portal 的侧栏

侧栏不是全量历史库，只承载当前工作记忆。

它应该展示：

- 当前活跃的 `Folio`
- 少量最近活跃的 `Slip`
- 待你查看、待你处理的对象

它不应该展示：

- 全量历史
- 无穷展开的层级树
- 一长串无筛选的旧对象

搜索、归档和 `Folio` 内完整列表承担历史检索。

侧栏在动态上也要遵守 Apple app 式层级切换，不应像树状文件管理器那样长期暴露复杂层级。

### 2.11 Portal 的页面骨架

#### `Slip` 页

固定为：

1. 标题区
2. DAG / plan card
3. 统一对话流（承载所有交互 + 内嵌 Deed 执行块）
4. 底部输入区 + 动作按钮区（按触发类型动态显示，见 §2.3）

动态要求：

- 进入 `Slip` 页要像进入一份内容对象，而不是打开任务详情
- 对话流中旧 Deed 执行块逐渐淡去（保质期机制）
- plan card / DAG 更新时有连续过渡

#### `Deed` 执行块（内嵌于 Slip 页）

Deed 不是独立页面，而是 Slip 对话流中的内嵌可展开块。执行后自动出现在对话流中。

展开内容：

1. DAG 进度（当前 Deed 在 DAG 上的流动）
2. Offering 版本和执行细节
3. 开始/停止（toggle）+ 收束按钮

动态要求：

- DAG 上 Move 的流动要有连续进度感
- 收束后执行块冻结为只读（产物标签 + 执行细节）

#### `Folio` 页

固定为：

1. 卷标题与摘要
2. 卷内关系图 / 脉络图
3. 当前活跃 `Slip`
4. 最近 `Slip`
5. 最近结果

Folio 结构视图中，每张 Slip 旁带内联操作区（按触发类型动态显示）：

| Slip 触发类型 | 内联显示 |
|-------------|---------|
| 手动触发 | 「执行」按钮（原子操作，创建 Deed + 立即运行） |
| 定时触发 | 下次触发时间 |
| 前序事件触发 | 等待条件状态（满足/未满足） |

已有 non-closed Deed 的 Slip → 按钮变为"执行中"状态（disabled）。Folio 结构视图是操作面，不只是导航列表。

动态要求：

- 进入 `Folio` 页更像打开一卷，而不是打开列表页面
- 卷中的 `Slip` 被拖动、吸附、收纳时，要有明显的对象感和物感
- 卷页里的对象组织优先于按钮控制

#### `Draft` 表现

`Draft` 应被看见为“尚未成札”的对象，而不是正式 `Slip` 的弱化版。

### 2.12 Portal 路由

Portal 的公开路由使用 `slug`：

- `/portal/`
- `/portal/slips/{slug}`
- `/portal/slips/{slug}/deeds/{deed_id}`（深链接，打开 Slip 页并滚动到对应 Deed 执行块）
- `/portal/folios/{slug}`

标题允许重复；公开地址由唯一 `slug` 承担。

## 3. Console

### 3.1 Console 的角色

Console 是维护者的治理面，不是主人的内容面。

它负责：

- 看系统状态
- 看对象结构
- 做受控的运维动作
- 查词典、查踪迹、查规则、查执行

它不负责：

- 与主人一样写作或协作
- 暴露原始实现细节
- 成为开发者调试后台

Console 的视觉和动态仍应延续 Claude + Apple 的组合，但要明确避免“文件管理器感”。

### 3.2 Console 的正式面板

Console 的正式面板为：

- `总览`
- `卷`
- `签札`
- `行事`
- `成文`
- `例行`
- `踪迹`
- `模型`
- `随从`
- `技能`
- `配给`
- `系统`
- `词典`

这些面板共同覆盖同一套正式对象和治理对象。

### 3.3 Console 的对象原则

Console 可以比 Portal 更明确地展示系统对象，但不能创造第二套机制。

因此：

- `卷` 面板看 `Folio`
- `签札` 面板看 `Slip`
- `行事` 面板看 `Deed`
- `成文` 面板看 `Writ`

这些对象之间应能互相跳转，看到的是同一现实，而不是不同模块各自维护的数据副本。

### 3.4 Console 的交互语法

Console 同样服从统一交互机制：

- 点开对象 = 进入对象详情
- 下拉 = 刷新当前层
- 返回 = 回上一层
- 长按 / 拖起 = 进入对象操作态

Console 可以更结构化，但不应引入另一套手势和另一套状态语义。

补充要求：

- Console 的进入、返回、详情展开、侧边细节、sheet 都应尽量贴近 Apple 原生 app 的层级动态
- 但 Console 不以拖拽整理对象为主语法
- Console 不做 Finder / Files 式的对象桌面

### 3.5 Console 的显示语言

Console 主显示使用**正式中文术语**。

英文 canonical name 只作为：

- 词典对照
- 调试辅助
- 字段说明

不再作为面板主标题或对象主显示名。

### 3.6 Console 的编辑原则

Console 默认只允许**结构化操作**，不允许随意编辑原始 JSON / Markdown / 文件内容。

明确例外：

- **技能正文编辑器**——技能正文本身就是文本能力对象，不适合强行结构化
- **Voice 文件编辑器**——Voice（Identity / Style）是自然语言画像文件，同样不适合强行结构化

### 3.7 Console 的隐私边界

维护者可以看到系统所需的状态、结构、健康与统计，但不应越过主人的内容边界。

Console 不应暴露：

- 主人的对话正文
- `Slip` 的私人内容全文
- `Offering` 正文
- `Vault` 正文
- 不必要的原始 Psyche 内容

Console 应优先呈现：

- 状态
- 结构
- 运行记录
- 规则
- 资源
- 版本与审计

## 4. Telegram

Telegram 的角色仍然是：

- 通知
- 极简命令

它不是第二个 Portal。

可承载：

- 完成通知
- 失败通知
- 状态查询
- 极简控制命令

不可承载：

- 主任务协作
- 大段反馈编辑
- 复杂对象组织

## 5. 风格要求

### 5.1 Portal 风格

Portal 要满足：

- 安静
- 克制
- 高排版权重
- 强对象感
- 强空间感
- 不面板化

并且必须同时成立：

- 静态尽量贴近 Claude 网页端
- 页面层级动态尽量贴近 Apple 原生 app
- 对象组织动态尽量贴近文件系统式对象操作

它的气质是“案桌上的草稿、签札与卷”，不是面板系统。

### 5.2 Console 风格

Console 同样遵守静态克制、动态一致，但应更像系统治理面。

颜色允许存在，但只能是：

- 语义标注
- 风险提示
- 当前焦点

不允许把颜色变成大面积彩色后台。

补充：

- Console 的动态语言只取 Apple 原生 app 那一面
- 不取文件管理器式对象组织那一面
- 详情 drill-down、列表到详情、侧边细节、sheet 切换要像统一系统

### 5.3 Portal 与 Console 的共同要求

两端必须共用：

- 同一对象语义
- 同一状态意义
- 同一手势语言
- 同一审美底色

它们 persona 不同，但不能长成两套产品。

更具体地说：

- Portal 与 Console 共用一套静态秩序
- Portal 与 Console 共用一套层级动态
- 只有 Portal 额外承担文件系统式对象组织
- Console 不承担这一层对象拖拽语法

## 6. 当前禁止事项

以下做法在 Portal 和 Console 中都不允许：

1. 用并列任务类型重新拆出多套前台物种
2. 继续把 `Deed` 视为任务本体
3. 用密集按钮阵列表达主要控制逻辑
4. 用原始 JSON / Markdown / 文件路径作为主要治理界面
5. 让同一种手势在不同地方表达不同类型的意图
6. 让 Portal 与 Console 维护两套不同对象现实
7. 在对话框中保留 slash command（已取消，所有操作由按钮/直接操作承担）
8. 让同一张 Slip 同时具有多种触发类型（手动/定时/前序事件互斥）
9. 把 Writ 排序当作建议——Writ 是强约束，前序未满足时按钮禁用
